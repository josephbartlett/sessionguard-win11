# SessionGuard 1.1.3

Patch release focused on fixing the combined installer so successful installs are not misreported as failures when Windows blocks the immediate tray-app launch.

## Highlights

- fixed the combined installer so a blocked post-install tray launch no longer makes the install look failed
- changed the immediate post-install app launch path to prefer the interactive shell context
- added better diagnostics for post-install app launch attempts and warnings
- clarified the install docs so a blocked immediate launch is explained as a non-fatal condition

## Included in this release

- new helper logic for best-effort installed app launch in the signed-in desktop session
- updated combined installer output fields for launch attempt, method, success, and warning text
- regression coverage for the new launch-failure handling path
- updated README, getting-started guidance, and generated setup README text

## Architecture

- `SessionGuard.Service` install behavior is unchanged
- `SessionGuard.App` still owns the tray icon and user-facing shell
- the combined installer now treats app auto-launch as best-effort instead of a required step for install success

## Limitations

- immediate tray launch still depends on the current Windows desktop session
- the combined install continues to register tray startup for the current user only
- SessionGuard remains a direct-download Windows utility rather than a Microsoft Store package
