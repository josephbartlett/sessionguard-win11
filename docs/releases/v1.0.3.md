# SessionGuard 1.0.3

Security, packaging, and startup hardening patch for the `1.0.x` line.

## Highlights

- service-backed mitigation and approval writes now require an authorized elevated app session
- corrupt local approval-state files no longer break scans or service initialization
- public release assets no longer preserve machine-local logs or state from reused publish directories
- sign-in startup no longer flashes the dashboard before hiding to the tray
- installed service status now follows the active install root and stays empty instead of falling back to stale repo-local publish metadata when nothing is installed
- combined uninstall now stops the running tray app before removing installed files

## Included in this release

- new command-authorization path for the named-pipe service control plane
- new structured `CanPerformServiceWrites` status flag shared across service, app, and smoke scenarios
- corrupt approval-file quarantine under `state/corrupt/`
- safer publish defaults for app, service, and combined bundle output
- updated runtime, getting-started, validation, and repo-operations documentation

## Architecture

- the service still owns machine-level mitigation and approval writes, but the pipe contract now distinguishes connectivity from caller authorization
- the desktop app now renders service connectivity and write access as separate states
- release packaging preserves operator config while stripping live runtime logs and state from public artifacts

## Limitations

- service-backed writes now require running `SessionGuard.App.exe` as administrator even when the service is already connected
- tray auto-start is still registered for the current user only
- the full product remains best suited to direct-download distribution rather than Microsoft Store packaging
