# SessionGuard 1.1.1

Patch release focused on tightening the tray-first runtime and clarifying the supported operator path for installed, elevated, and source-run sessions.

## Highlights

- fixed single-instance scoping so installed, source-run, and elevated app sessions no longer collide
- clarified the elevated app flow for service-owned guard-mode, mitigation, and approval actions
- renamed the preferred public release asset to the setup zip to make the intended end-user download clearer
- aligned release workflow asset ordering and install docs with the new setup-first distribution story

## Included in this release

- updated app-instance coordination to scope by user, executable path, and elevation level
- added regression tests for instance scoping and normalized path handling
- updated the bundled install README and repo docs to document `-DoNotLaunchApp`, `-DoNotStartService`, and the elevated app path
- updated release packaging metadata and tests so current-version fixtures no longer rely on stale `1.0.4` values

## Architecture

- `SessionGuard.Service` still owns machine-level monitoring and state-changing actions
- `SessionGuard.App` now distinguishes standard and elevated sessions more cleanly when enforcing single-instance behavior
- release automation still publishes app, service, setup, and source assets, but now presents the setup asset as the intended first choice

## Limitations

- service-backed writes still require both service connectivity and an elevated app session
- tray auto-start remains user-scoped through the current user's Run key
- SessionGuard remains a direct-download Windows utility rather than a Microsoft Store package
