# SessionGuard 1.0.4

Security, packaging, and install-flow hardening patch for the `1.0.x` line.

## Highlights

- guard-mode changes now require the same elevated service-backed path as mitigation and approval writes
- published app, service, and bundle manifests no longer include machine-specific absolute paths
- combined install now launches the installed tray app unless that exact installed executable is already running
- the combined bundle now ships an install-focused README instead of the full source-repo README

## Included in this release

- updated named-pipe authorization for service-backed guard-mode changes
- clearer app-side permission handling for connected but non-elevated sessions
- runtime-neutral manifest generation for app, service, and bundle publish outputs
- improved combined install process detection for installed tray-app launch
- install-focused bundle README generation for end-user release bundles
- expanded authorization and service-script test coverage

## Architecture

- the service remains the authoritative runtime for state-changing guard, mitigation, and approval actions
- the app continues to separate service connectivity from write access in the operator experience
- published bundle metadata now describes the packaged runtime without leaking build-machine filesystem details

## Limitations

- service-backed writes still require running `SessionGuard.App.exe` as administrator even when the service is installed and connected
- tray auto-start is still registered for the current user only
- SessionGuard remains a direct-download Windows utility rather than a Microsoft Store package

## Verification

- `dotnet build SessionGuard.sln`
- `dotnet test SessionGuard.sln`
- `powershell -ExecutionPolicy Bypass -File scripts/ci/Invoke-WindowsValidation.ps1`
- `powershell -ExecutionPolicy Bypass -File scripts/release/Publish-SessionGuardReleaseAssets.ps1 -Version 1.0.4 -Configuration Release -Runtime win-x64 -SelfContained`
