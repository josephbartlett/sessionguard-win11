# SessionGuard 1.2.0

Tray-workflow refinement release focused on making the tray menu the default day-to-day control surface while keeping the full dashboard and elevated write path clear when you need them.

## Highlights

- refined the tray menu into a shorter daily workflow with status, next step, one common action, and a smaller support area
- added `Open elevated controls` as the explicit path for service-backed write actions when the current app session is still read-only
- aligned tray, dashboard, and notification wording around the same service-versus-write-access model
- added a `--technical-view` startup path so the elevated SessionGuard window can open directly into the detailed control surface without creating another tray shell

## Included in this release

- updated tray menu layout in `SessionGuard.App`
- updated operator alert wording and transition messaging in `SessionGuard.Core`
- updated dashboard guidance and technical-view actions in `SessionGuard.App`
- updated the bundled setup README plus operator docs in:
  - `README.md`
  - `docs/getting-started.md`
  - `docs/runtime-model.md`
  - `docs/manual-validation.md`
- updated `docs/plans/v1.2.0-everyday-tray-workflow-plan.md` with the canonical action map

## Architecture

- `SessionGuard.Service` still owns background monitoring and state-changing actions
- `SessionGuard.App` remains the tray and dashboard shell, but the tray menu is now more intentionally the primary day-to-day workflow
- the elevated controls path stays inside the same app/runtime model by launching a separate elevated no-tray SessionGuard window

## Limitations

- service-backed writes still require both service connectivity and an elevated SessionGuard window
- the tray shell is calmer, but deep diagnosis still depends on opening the main window
- SessionGuard remains a direct-download Windows utility rather than a Microsoft Store package
