# SessionGuard 0.4.1

UI Smoke Automation release focused on making the WPF shell launchable, inspectable, and reviewable through deterministic local automation rather than ad hoc manual startup.

## Highlights

- added deterministic UI scenarios that drive the real WPF app into known states for automation and review
- added a repo-owned UI smoke runner that launches the app, checks key controls through Windows UI Automation, and captures screenshots
- added stable automation IDs to the dashboard so UI assertions stay durable as the app evolves
- fixed visible dashboard issues exposed by the smoke run, including clipped text and raw enum labels in the workspace table

## Included in this release

- new `tests/SessionGuard.UiSmoke` for smoke execution and screenshot capture
- new `scripts/ui/Run-UiSmoke.ps1` entry point for repeatable local UI verification
- scenario-backed app startup wiring in `SessionGuard.App` for deterministic render states
- screenshot outputs under `artifacts/ui/smoke/` for visual inspection and release validation
- updated docs describing the smoke flow and review expectations

## Architecture

- `SessionGuard.Core` now carries a shared UI smoke scenario catalog and automation IDs
- `SessionGuard.App` can bootstrap against a scenario-backed control plane for deterministic UI rendering
- `SessionGuard.UiSmoke` is a dedicated desktop automation runner instead of overloading unit-test infrastructure
- the smoke flow complements the service/control-plane architecture rather than replacing it

## Limitations

- the smoke run validates window launch, control presence, and visible layout regressions, but it is not a full end-to-end operator workflow test
- screenshot review is still heuristic and may not catch every styling regression
- UI Automation coverage is currently Windows-local only and is not yet wired into CI
