# v1.2.0 Everyday Tray Workflow Plan

Status:
Implemented locally and pending final user confirmation before push.

## Goal

Make SessionGuard feel better in daily use when the user mostly interacts through the tray icon and only opens the dashboard when they need detail.

The core service, policy, and install baseline is already in place. `v1.2.0` should improve how clearly SessionGuard communicates state and what action, if any, the user should take next.

## Product outcome

When this plan is complete, the expected day-to-day experience should be:

- the tray menu explains the current state in plain language without opening the dashboard
- the most common actions are available in one obvious place
- notifications are rarer, clearer, and less repetitive
- the dashboard stays useful, but it no longer feels like the primary control surface for common cases
- service-connected versus elevated-action requirements are obvious instead of implied

## Guiding constraints

- preserve the current service-owned boundary for guard mode, mitigation, and approval writes
- do not add more surface area unless it removes confusion from the existing surface area
- prefer fewer tray actions with clearer wording over more tray actions with technical wording
- keep the main dashboard quieter, but do not hide important diagnostic state from users who intentionally open it
- maintain deterministic behavior for startup, tray state, and notifications so validation remains automation-friendly

## Phase 1 - Runtime and workflow audit

Purpose:
Understand the current tray-first behavior before changing it.

Implementation targets:
- inventory all current tray menu items, notifications, dashboard actions, and repeated status copy
- document which actions are currently exposed in:
  - tray menu
  - dashboard overview
  - technical view
  - notifications
- map each action to one canonical home:
  - tray-primary
  - dashboard-only
  - notification-driven
- identify every state where the app currently says:
  - service unavailable
  - service connected but elevated app required
  - approval required
  - approval active
  - restart blocked

Acceptance criteria:
- there is a written inventory of the current tray/dashboard/notification action model
- duplicated actions and duplicated status copy are identified explicitly
- there is a proposed canonical home for each common action before UI changes begin

Audit results:

- tray menu before implementation exposed summary, next step, context, primary action, open dashboard, open technical view, scan now, status details, Windows Update options, and exit
- dashboard overview already had a contextual action card, but service-connected versus elevated-action requirements were described differently from the tray wording
- technical view owned mitigation, approval, config, policy, and log actions, but the elevated path was still described as a manual relaunch instead of a first-class workflow
- notifications already triggered on transitions, but the tray and dashboard were still duplicating some of the same operator wording without a single canonical action map

Canonical action map:

- `Scan now`: header utility and support-only tray action
- `Open elevated controls`: contextual primary action when the service is connected and a write action is relevant, plus a technical-view support action
- `Grant approval window`: contextual primary action when approval is required and write access is available
- `Apply protections`: contextual primary action when mitigations are available and write access is available
- `Open dashboard`: stable tray action for fuller plain-language context
- `Open technical details`: stable tray action for the diagnostic view
- `Windows Update options`: support-only action when SessionGuard cannot legally perform the change itself
- `Open config`, `Open policies`, `Open logs`: technical-view only

## Phase 2 - Tray menu refinement

Purpose:
Make the tray menu the clearest day-to-day status surface.

Implementation targets:
- reorganize the tray menu into a stable order:
  - current status
  - next step
  - common action
  - open dashboard / open technical view
  - advanced/support items
- tighten wording for the top tray summary so it answers:
  - what is happening
  - whether action is needed
  - whether that action requires elevation
- reduce duplicated or low-value tray entries
- ensure tray actions do not expose service-owned writes when the current session cannot legally perform them

Acceptance criteria:
- a user can understand the current state from the tray menu alone in the common cases
- the tray menu has a clear action hierarchy instead of feeling like a mirror of the dashboard
- the tray menu clearly distinguishes read-only status from actionable status

## Phase 3 - Notification refinement

Purpose:
Make notifications helpful rather than noisy.

Implementation targets:
- review each notification trigger and classify it as:
  - must notify
  - optional notify
  - should be silent
- tighten suppression and deduplication rules for repeated stable states
- improve wording for:
  - restart pending
  - approval required
  - approval expiring soon
  - service unavailable
  - service recovered
  - action requires elevated app
- decide which notifications should push the user toward:
  - tray-only action
  - opening the dashboard
  - relaunching the app as administrator

Acceptance criteria:
- repeated scans in the same state do not create repeated notifications
- notifications tell the user what changed and what to do next
- notifications no longer compete with the tray menu for the same message

## Phase 4 - Dashboard and tray consistency

Purpose:
Keep the dashboard aligned with the tray instead of making them feel like separate products.

Implementation targets:
- unify the status labels used by:
  - tray summary
  - overview card
  - notification text
  - logs where operator-facing copy leaks into the UI
- review the main dashboard actions and move any low-value repeated action out of the overview if the tray now owns it
- tighten the language around service connectivity versus write access
- keep `Simple view` calm and plain-language while letting `Technical view` retain the diagnostic depth

Acceptance criteria:
- tray, overview, and notifications use the same state vocabulary
- the dashboard overview does not reintroduce the “button pile” problem
- the app no longer implies that service connectivity and write access are the same thing

## Phase 5 - Validation and release hardening

Purpose:
Make the tray-workflow refinement safe to ship.

Implementation targets:
- expand automation coverage for tray-oriented state mapping where practical
- add tests for:
  - notification deduplication
  - tray action availability by state
  - service-connected versus elevated-action wording
- refresh UI smoke expectations for any overview changes
- perform a screenshot review of the updated calm-state and action-needed states
- verify installed startup, tray reuse, and elevated-action behavior still work after the tray/menu changes

Acceptance criteria:
- the updated tray workflow is validated by both tests and screenshot review
- the installed app still starts quietly and reuses the running tray process
- the release candidate does not regress action gating or installed startup behavior

## Required code review passes

This release should include at least three deliberate review passes over the whole codebase, not just the files changed for the tray polish.

### Pass 1 - Runtime and security audit

Review focus:
- service/app boundaries
- tray actions versus service-owned write actions
- startup and relaunch behavior
- elevation checks and authorization messaging

Expected output:
- fixes for any accidental boundary regression or misleading action availability

### Pass 2 - UX and state-model audit

Review focus:
- duplicated wording
- inconsistent state labels
- confusing tray action placement
- noisy or repetitive notification triggers

Expected output:
- cleaner copy
- fewer duplicate actions
- clearer user-facing state mapping

### Pass 3 - Release and documentation audit

Review focus:
- install and startup docs
- tray behavior docs
- screenshots and manual validation steps
- release packaging and release-note wording

Expected output:
- aligned docs
- supportable release messaging
- no stale dashboard-first assumptions in the shipped docs

## Testing requirements

The release is not complete without all of the following:

- `dotnet build SessionGuard.sln`
- `dotnet test SessionGuard.sln`
- `powershell -ExecutionPolicy Bypass -File scripts/ui/Run-UiSmoke.ps1`
- `powershell -ExecutionPolicy Bypass -File scripts/ci/Invoke-WindowsValidation.ps1`
- screenshot review of calm, restart-pending, protected-workspace, and local-fallback states
- manual validation for:
  - app-only mode
  - app plus console-hosted service mode
  - installed service mode
  - non-elevated app behavior
  - elevated app behavior for service-owned writes
  - tray reuse and repeated launches

Likely live checks:

- install the current setup bundle
- validate tray behavior after sign-in style startup
- validate that the elevated app path remains separate and intentional
- validate that repeated status scans do not spam notifications

## Suggested execution order

1. complete Phase 1 workflow audit and define the canonical action map
2. implement Phase 2 tray menu refinement
3. implement Phase 3 notification refinement
4. complete Pass 1 full-code audit
5. implement Phase 4 dashboard and tray consistency cleanup
6. complete Pass 2 full-code audit
7. complete Phase 5 validation and screenshot review
8. complete Pass 3 full-code audit
9. prepare release notes and package `v1.2.0`

## Out of scope for v1.2.0

- changing UI frameworks
- major restart-signal provider expansion
- kernel or enterprise policy features
- Microsoft Store packaging for the full service-backed build
- full signing and installer overhaul beyond what is needed to keep direct-download distribution understandable
