# v0.4.0 Workspace Safety Plan

## Goal

Move SessionGuard from process-presence awareness to a more useful model of workspace disruption risk without overstating certainty or attempting full recovery too early.

## Guiding constraints

- stay read-only for the first snapshot pass
- keep heuristics explicit and explainable
- do not claim knowledge of unsaved user state unless a signal truly supports it
- preserve the current service and dashboard architecture rather than bypassing it

## Phase 1 - Workspace model and heuristics

Purpose:
Define the data model for workspace risk and create the first pluggable heuristics layer.

Implementation targets:
- add a workspace-state subsystem in `SessionGuard.Core`
- distinguish generic protected processes from higher-risk workspace signals
- add heuristics for:
  - terminals and shells
  - editor and IDE sessions
  - browsers
  - local dev-server style processes
- record why a workspace was considered risky

Acceptance criteria:
- the app can show workspace-risk reasons, not just process names
- heuristics can be extended without changing restart-signal providers
- tests exist for the first heuristic rules

## Phase 2 - Snapshot metadata foundation

Purpose:
Persist bounded, machine-readable workspace metadata without attempting recovery yet.

Implementation targets:
- create snapshot metadata models in `SessionGuard.Core`
- persist snapshot metadata under `state/`
- capture:
  - detected process groups
  - workspace-risk reasons
  - timestamps
  - confidence level
- keep snapshot output advisory and local only

Acceptance criteria:
- a snapshot metadata file is written during scans when workspace risk is present
- snapshot persistence does not block or destabilize the main scan loop
- docs explain exactly what the snapshot does and does not contain

## Phase 3 - Dashboard and service integration

Purpose:
Make workspace-risk information visible in both the service-backed and local-fallback paths.

Implementation targets:
- surface workspace-risk summary in the dashboard
- add per-workspace or per-group explanation text
- propagate snapshot and workspace-risk state through the control plane
- keep the service snapshot authoritative when the app is attached to the service

Acceptance criteria:
- the dashboard explains why a session is considered risky
- service-backed mode and local fallback mode render compatible workspace summaries
- UI copy remains honest about inference versus certainty

## Phase 4 - Validation and release hardening

Purpose:
Keep the new heuristics from becoming hand-wavy or fragile.

Implementation targets:
- add regression tests for heuristic output
- review false-positive cases for browsers, terminals, and editors
- update `docs/limitations.md` with any newly discovered confidence boundaries
- prepare release notes and manual validation steps for `v0.4.0`

Acceptance criteria:
- heuristic behavior is covered by automated tests
- documentation names the main false-positive and false-negative risks
- the release remains explicit that SessionGuard mitigates disruption rather than guaranteeing recovery

Current branch status:
- completed
- automated regression coverage now includes heuristic boundary cases and snapshot persistence
- release notes, manual validation guidance, and packaging defaults have been prepared for `v0.4.0`

## Dependencies and sequencing

Recommended order:

1. finish the remaining `0.3.x` stabilization work
2. ship Phase 1 workspace models and heuristic interfaces
3. add Phase 2 snapshot persistence
4. wire Phase 3 into the service and dashboard
5. complete Phase 4 validation before tagging `v0.4.0`

## Out of scope for v0.4.0

- automatic application relaunch
- browser-tab or editor-buffer recovery
- approval workflows
- enterprise policy orchestration
- any claim of reliable full-session restoration
