# v1.1.0 Tray-First Experience Plan

## Goal

Make SessionGuard feel like a calm, always-available Windows utility instead of a dashboard that happens to have a tray icon.

The next app milestone should reduce window dependence, make the tray path the normal daily workflow, and keep the full technical view available without forcing it into the foreground.

## Product outcome

When this plan is complete, the expected day-to-day experience should be:

- the service starts with Windows when installed
- the app starts at sign-in, stays quiet, and lives in the tray by default
- the tray surface shows the current risk in plain language
- common actions do not require reopening the full dashboard
- the full window still exists for diagnosis, policy review, and advanced controls

## Guiding constraints

- preserve the current service-owned boundary for guard-mode, mitigation, and approval writes
- do not hide important limitations behind a simplified tray flow
- prefer fewer surfaces and clearer action prompts over adding more controls
- keep the app usable for non-technical users in `Simple view` without removing the `Technical view`
- keep all startup, tray, and notification behavior deterministic enough for automated validation

## Phase 1 - Runtime and UX contract

Purpose:
Define the tray-first behavior before changing UI structure.

Implementation targets:
- define the default launch behavior for:
  - first run
  - sign-in auto-start
  - manual app launch when the service is already running
  - local fallback when the service is unavailable
- define the minimum tray surface:
  - current status
  - one plain-language summary
  - one primary action
  - access to the full window
- define which current dashboard actions move into:
  - tray menu
  - contextual notification prompts
  - full window only
- document which actions still require:
  - service connectivity
  - elevated app execution

Acceptance criteria:
- there is a written runtime contract for app startup, tray ownership, and action ownership
- a user can answer "what runs at boot, what runs at sign-in, and what owns the tray" from the docs alone
- the allowed tray actions are intentionally limited rather than mirroring the whole dashboard

## Phase 2 - Tray shell redesign

Purpose:
Make the tray path the primary operator surface.

Implementation targets:
- redesign the tray menu into a compact status-and-action shell
- add clearer state buckets such as:
  - no action needed
  - save work soon
  - approval needed
  - service attention needed
- keep the full dashboard accessible as:
  - open dashboard
  - open technical view
- reduce duplicate controls between tray, dashboard overview, and notifications
- ensure the dashboard remains useful, but no longer feels like the mandatory daily entry point

Acceptance criteria:
- the tray menu communicates current state without requiring the main window
- the most common action can be taken from the tray when platform boundaries allow it
- the dashboard overview is quieter because it no longer needs to expose every control all the time

## Phase 3 - Notification and prompt cleanup

Purpose:
Make notifications useful, rare, and action-oriented.

Implementation targets:
- tighten notification thresholds so SessionGuard does not notify on low-value state churn
- improve notification copy for:
  - restart pending
  - approval active
  - approval expiring soon
  - service unavailable
  - service recovered
- decide when a notification should point to:
  - tray-only action
  - opening the main window
  - an elevated relaunch instruction
- review all operator copy for plain-language clarity

Acceptance criteria:
- notifications explain what changed and what the user should do next
- the app does not emit duplicate or noisy notifications for the same stable state
- non-technical users can tell whether SessionGuard is informing, warning, or asking for action

## Phase 4 - Startup and install polish

Purpose:
Make the installed experience feel coherent.

Implementation targets:
- verify the sign-in startup path really behaves as tray-first in installed mode
- tighten startup arguments and restore behavior so manual launches and sign-in launches do not fight each other
- review the combined install and uninstall flow for:
  - tray startup registration
  - already-running installed app detection
  - startup behavior after upgrade
- document the installed experience in one short path for non-technical users

Acceptance criteria:
- installed mode starts quietly and predictably
- manual launch, sign-in launch, and service startup do not create confusing duplicate windows or processes
- install and uninstall docs match the real product behavior exactly

## Phase 5 - Validation and release hardening

Purpose:
Keep the tray-first shift from becoming a fragile UX-only refactor.

Implementation targets:
- expand UI smoke automation to cover:
  - tray-first startup
  - current-status tray text
  - approval-needed state
  - service-unavailable tray behavior
  - opening the full dashboard from the tray path
- add tests for notification suppression and repeated-state deduplication
- add tests for tray-menu action availability based on:
  - service connectivity
  - elevation
  - policy state
- perform explicit visual review of generated screenshots

Acceptance criteria:
- the tray-first flows are automation-covered, not just manually observed
- release validation proves the app still launches, stays quiet at sign-in, and exposes the expected tray controls
- docs and screenshots are updated in the same release as the behavior change

## Required code review passes

This milestone must include at least three deliberate review passes over the codebase, not just the files edited during implementation.

### Pass 1 - Runtime and security audit

Review focus:
- startup flow
- tray ownership
- service-app boundaries
- elevation requirements
- authorization and action gating

Expected output:
- a list of any boundary regressions or write paths accidentally exposed from the tray flow
- fixes before the tray redesign is considered stable

### Pass 2 - UX and state-model audit

Review focus:
- duplicated state explanations
- inconsistent status mapping between tray, dashboard, notifications, and logs
- confusing or overly technical wording
- state transitions that could create notification spam or operator confusion

Expected output:
- copy and behavior cleanup
- one source of truth for operator-facing state labels where practical

### Pass 3 - Release and documentation audit

Review focus:
- install and upgrade path behavior
- release bundle contents
- startup registration docs
- screenshots and validation docs
- stale references to old dashboard-first assumptions

Expected output:
- cleaned docs
- corrected release packaging if needed
- a release candidate that is supportable without verbal explanation

## Testing requirements

The milestone is not complete without all of the following:

- `dotnet build SessionGuard.sln`
- `dotnet test SessionGuard.sln`
- `powershell -ExecutionPolicy Bypass -File scripts/ui/Run-UiSmoke.ps1`
- `powershell -ExecutionPolicy Bypass -File scripts/ci/Invoke-WindowsValidation.ps1`
- explicit screenshot review of tray-first UI smoke output
- manual validation for:
  - app-only mode
  - app plus console-hosted service mode
  - installed service mode
  - sign-in style startup arguments
  - non-elevated behavior
  - elevated behavior for service-owned actions

Likely elevation-required checks:
- installed service validation
- service-backed action validation
- startup registration and install or uninstall checks

## Suggested execution order

1. complete Phase 1 runtime contract and simplify the action model
2. implement Phase 2 tray shell redesign
3. implement Phase 3 notification cleanup
4. complete Pass 1 full-code audit
5. implement Phase 4 startup and install polish
6. complete Pass 2 full-code audit
7. expand Phase 5 automation and manual validation
8. complete Pass 3 full-code audit
9. prepare release notes, screenshots, and updated operator docs for `v1.1.0`

## Out of scope for v1.1.0

- replacing the WPF shell with a new UI framework
- full workspace recovery or app relaunch orchestration
- centralized policy management
- Microsoft Store packaging for the full service-backed product
- major restart-signal provider expansion unrelated to tray-first UX
